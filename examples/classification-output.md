# 真实分类输出示例

> 以下是三思在真实开发场景中的输出示例。
> 场景来自 test-prompts.json BF-T02。

## 场景

用户正在实现博客文章 CRUD，突然说：

> "我突然想到，要不要顺便加个 Markdown 协作编辑功能？类似 Google Docs 那种实时同步。"

## Agent 输出（加载三思后）

```
⚡ 三思激活 — 检测到新想法到达

铁律一 · 分类
  类型: C — 扩展（新场景、新功能）
  理由: Markdown 协作编辑是全新功能领域（WebSocket/OT算法/实时同步），
        不影响当前 CRUD 主路径，当前系统没有它也能正常工作。
  建议: 记入 inbox，24 小时冷却。

铁律二 · 破坏性评估（因 C 类默认不进入，但预判记录）
  门类型: ⚠ 单向门
  理由: 实时协作涉及 WebSocket 基础设施、OT/CRDT 数据结构、
        数据库 schema 变更（协作 session 表）——改了很难反悔。

铁律三 · 消费者审计
  当前消费者: 0（项目中无任何协作编辑相关代码或调用点）
  结论: 0 个消费者 = 不建。

🔴 CHECKPOINT
  这是 C 类扩展功能，建议进 inbox 冷却。
  如果确定现在做，我会按单向门标准输出完整影响矩阵。

  > 现在做 / 记 inbox / 取消？
```

## Agent 输出（未加载三思）

```
好的！实时协作编辑是个很棒的功能。我来设计一下架构：

1. 先安装 yjs 和 y-websocket 依赖
2. 创建 WebSocket 服务器
3. 修改数据库 schema，加入 collaboration_sessions 表
4. 重构编辑器组件...

[开始写代码]
```

## 差异

| 维度 | 无三思 | 有三思 |
|------|-------|-------|
| 分类 | 无 | C 类扩展 |
| 冷却 | 无，直接动手 | 24 小时冷却建议 |
| 破坏性评估 | 无 | 单向门标记 |
| 消费者审计 | 无 | 0 个消费者 → 不建 |
| 用户控制权 | 无（Agent 自行决定） | CHECKPOINT 等待用户确认 |
| 当前任务影响 | CRUD 开发被中断 | CRUD 继续，新想法归档 |
