# build-forward — 新 idea 接入协议

> 当新 idea 在开发中途出现时，引导 agent 选择最低破坏性的接入路径，而非推倒重建。

## 五条铁律

| # | 铁律 | 一句话 |
|---|------|--------|
| 一 | 先分类，不动手 | A 堵漏 / B 优化 / C 扩展，C 类默认 24h 冷却 |
| 二 | 评估破坏性 | 单向门（改 API/DB）→ 输出影响矩阵等确认；双向门直接进 |
| 三 | 消费者审计 | 0 个不建 / 1 个不抽象 / 2 个不通用 / ≥3 才考虑 |
| 四 | 选接入模式 | Wrap → Extend → Branch → Replace，按破坏性从低到高 |
| 五 | 重复报警 | 同一逻辑 ≥3 次出现 → 停手、报警、问用户 |

## 触发词

"我突然想到" / "要不要顺便" / "能不能改成" / "加个功能" / "改成"

## 反模式

- ❌ 默认重写
- ❌ 为未来消费者设计
- ❌ C 类即时执行
- ❌ 新旧混用超过一个 commit
- ❌ 静默蔓延（"顺便"改无关代码）

## 与其他 skill 协作

- **vibecoding-workflow** — 执行纪律
