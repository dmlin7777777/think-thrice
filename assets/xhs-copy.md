## 版本一：痛点共鸣型（推荐首发）

### 标题
🤯 用 AI Agent 写代码的人，都崩溃过这一刻。

### 正文
不知道你有没有过这种经历——

用 Claude Code 或者 Cursor 写代码，开发正顺畅的时候，脑子里突然蹦出一个新想法："诶，要不要顺便加个实时协作？"

然后你的 Agent 说："好主意！"

接着它开始重写你的认证层、改你的数据库 schema、把三个已经跑通的接口改得面目全非。

你喊停，已经晚了。3 个文件没了。

我上周刚经历完，忍不了。于是写了一个 Skill。

**build-forward —— Agent 的新想法防火墙。**

加载之后，每次你说"我突然想到…"，Agent 不再直接推倒重写，而是：

✅ 先分类（是新功能？还是优化？还是补漏？）
✅ 评估破坏性（是单向门还是双向门？改 DB schema = 单向门，暂停等确认）
✅ 审计消费者（0 个调用方？不建。1 个？内联。≥3 个？才考虑抽象）
✅ 选最低破坏路径（Wrap → Extend → Branch → Replace，Replace 是最后手段）
✅ 重复报警（同一模式 ≥3 次？停下问你要不要统一处理）

**本质是五条铁律，一条都不能跳。**

🔗 GitHub: dmlin7777777/build-forward
📦 一行安装：`npx skills add dmlin7777777/build-forward`

兼容 Claude Code、WorkBuddy、Cursor 等主流 Agent 环境。

## 版本二：硬核技术型

### 标题
🛡️ 给你的 AI Agent 装一个"防火墙"｜开源 Skill 推荐

### 正文
跟 AI Agent 协作开发，最大的坑不是 AI 不够聪明——是它太"积极"了。

你说一句"我想加个功能"，它立刻重写半个项目。
这是在 vibe coding 里最常见也最贵的 bug。

我写了一个 Agent Skill 来解决这个问题：

**build-forward · 五铁律门控协议**

🔹 铁律一：分类优先，不动代码
紧急修复（A）/ 体验优化（B）/ 新功能（C）→ 不同路线，不同处置

🔹 铁律二：评估破坏性
单向门（DB/API/全局状态）→ 🛑 CHECKPOINT，等用户确认
双向门（UI/新字段/新路由）→ 可以推进

🔹 铁律三：消费者审计
0 调用方不建，1 调用方内联，2 才提取，≥3 考虑抽象层
——从根上扼杀"为未来而建"

🔹 铁律四：选最低破坏路径
Wrap → Extend → Branch → Replace（最后手段）

🔹 铁律五：重复报警
同一逻辑 ≥3 处重复 → 🛑 CHECKPOINT

**加载前 vs 加载后**，Agent 行为判若两人。

📊 评分：81/100（鲁班工坊独立验收）
📄 协议：MIT 开源
⬇️ 安装：`npx skills add dmlin7777777/build-forward`

不只是一个 Skill，是一套给 Agent 的决策纪律。
