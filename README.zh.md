# build-forward

**一个让 AI Agent 在不推倒重建的前提下接入新想法的 Skill。**

> 当新想法在开发中途出现时，大多数 AI Agent 的默认反应是推倒重写。`build-forward` 给 Agent 一套协议，让产品向前生长而非原地推翻。

兼容 Claude Code、Cursor、Codex CLI、Gemini CLI、OpenClaw，以及任何支持 SKILL.md 的 Agent。

---

## 解决什么问题

在 AI 辅助开发中，想法来得比架构规划快。Agent 对"我想加个 X"的典型回应是把挡路的代码全部重写。

这造成一个痛苦的循环：

```
新想法 → Agent 重写基础 → 又来了个新想法 → Agent 又重写 → ……
```

现有的 Skill 如 `incremental-implementation` 解决了*已知需求如何增量实现*的问题，但没人回答：

> **当一个可能与当前正在构建的东西冲突的新想法出现时，你该怎么做？**

`build-forward` 填补了这个空白。

---

## 工作原理

五条铁律，每当新想法出现时按顺序执行。触发词包括"我突然想到" / "要不要顺便" / "能不能改成" / "加个功能"。

### 铁律一 — 先分类，不动手

任何代码写之前，先给想法贴标签：

| 类型 | 判断标准 | 默认处置 |
|------|----------|----------|
| **A — 堵漏** | 影响当前主路径，不修会出错 | 现在处理，进铁律二 |
| **B — 优化** | 体验更好，但当前能用 | 问用户：现在做还是记 inbox？ |
| **C — 扩展** | 新场景、新功能 | 默认记 inbox，24 小时冷却 |

C 类的 24 小时冷却防止冲动开发。大多数功能冲动要么消退，要么结晶为更清晰的需求。

**分类模糊时的处置：**

| 触发条件 | 一线修复 | 兜底 |
|----------|----------|------|
| A 还是 B 分不清 | 按 A 处理（堵漏优先，宁严勿松） | — |
| B 还是 C 分不清 | 问："这个功能不做的代价是什么？" | 用户说"不急"→ C；"影响体验"→ B |
| C 类但涉及安全 | 升级为 A，立即处理 | — |

### 铁律二 — 评估破坏性（单向门 vs 双向门）

动手之前，先判断改动性质：

**双向门**（可以反悔）→ 直接进铁律三：
- 改 UI 样式、文案、布局
- 加新字段（不删旧字段）
- 加新函数（不改旧函数签名）
- 加新路由（不改旧路由）

**单向门**（改了很难反悔）→ 🔴 CHECKPOINT · 🛑 STOP，必须先输出影响矩阵：

```
单向门影响评估
═══════════════════════════════════════

直接改动
  [文件路径] — [改动内容简述]
  [接口名称] — [签名变更说明]

下游影响
  [消费者1] — [影响方式]
  （grep 确认，不编造）

回滚成本：🔴高 / 🟡中 / 🟢低
  理由：[为什么是这个评级]

建议
  □ 现在做 — [理由]
  □ 开新分支 — [理由]
  □ 记 inbox — [触发条件]
  □ 不接 — [为什么当前方案已足够]
```

完整模板和回滚成本评级标准见 [`references/impact-matrix-template.md`](references/impact-matrix-template.md)。

### 铁律三 — 消费者审计（建之前先数使用者）

新建任何 API、模块、组件、工具函数之前，先问：

> "这个东西的消费者是谁？现在有几个实际调用点？"

| 消费者数量 | 处置 |
|-----------|------|
| **0 个** | 不建。记 inbox C 类，等真实需求出现 |
| **1 个** | 最简实现，写在调用处，不抽象 |
| **2 个** | 可以提取，但不做通用化设计 |
| **≥ 3 个** | 才考虑抽象层、接口、配置化 |

这条规则把"为将来扩展而设计"扼杀在摇篮里。

**消费者数量不确定时：**

| 触发条件 | 处置 |
|----------|------|
| 外部调用者（第三方 API 消费者） | 按 ≥3 保守评估，改动走单向门流程 |
| grep 搜不到但怀疑有隐式调用 | 标记为"消费者待确认"，先不改，等运行时报错再定位 |
| 跨仓库/跨项目调用 | 默认当消费者存在。**不删不改旧接口签名**，只追加 |

### 铁律四 — 选接入模式

选破坏性最低的路径：

| 模式 | 适用场景 | 做法 |
|------|----------|------|
| **Wrap（包一层）** | 新功能是对现有功能的增强 | 在现有函数/组件外加一层，旧路径不动 |
| **Extend（延伸）** | 新场景共享大部分逻辑 | 加新参数/分支/路由，不改已有路径 |
| **Branch（并行）** | 新旧路径差异大 | 并行实现，逐步迁移消费者 |
| **Replace（替换）** | 旧实现已是负债 | 先写新实现+测试，再删旧的，逐步迁移。**禁止**：新旧混用超过一个 commit |

### 铁律五 — 重复报警

同一段逻辑在不同地方出现 **≥ 3 次**：

1. 🔴 CHECKPOINT · 🛑 STOP：不要静默复制第 3 次。
2. 报警："我发现 [X 逻辑] 重复了 [N] 处，建议提取。现在统一还是记 inbox？"
3. 用户决定。无论哪种，记录在案。

1-2 次重复是正常的，3 次才是信号，不要为 2 次提前抽象。

## 防止的反模式

- ❌ **默认重写** — 看到新 idea 就重构现有代码，不先评估破坏性
- ❌ **为未来消费者设计** — 建造没有调用者的 API/模块
- ❌ **C 类即时执行** — 新功能冲动绕过 24 小时冷却直接开发
- ❌ **混合模式超期** — 新旧两套方案并存超过一个 commit
- ❌ **静默蔓延** — "来都来了，顺便把……"

## 与同类 Skill 的对比

| Skill | 触发点 | 核心问题 |
|-------|--------|----------|
| `incremental-implementation` (Addy Osmani) | 已知需求 | 如何稳步实现 |
| `vibecoding-workflow` | 每次开发任务 | 我在不在范围内？ |
| **`build-forward`** | **开发中途出现新想法** | **哪条路径破坏最小？** |

这几个 Skill 互补。`build-forward` 管想法出现的那一刻；接入路径确定后，`incremental-implementation` 接管实现阶段。
